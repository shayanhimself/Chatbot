# M0 — Scaffold Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Walking skeleton — every module exists thin, the toolchain is proven end to end (build → Hilt codegen → unit + Compose/Robolectric tests → app launches on emulator → journey passes).

**Architecture:** Multi-module Android app per the `architecture` project skill: `:app` (Hilt root, MainActivity), `:shared` (KMP library, `androidTarget()` only), `:core:ui` (M3 theme), four `:feature:*` stubs. No navigation, no networking, no persistence yet — those are M1.

**Tech Stack:** Kotlin 2.3.21, KSP 2.3.10, AGP 9.3.0, Gradle 9.6.1, Compose BOM 2026.06.01, Hilt 2.60.1, JUnit4, Robolectric 4.16.1, Compose testing v2, `android` CLI for project creation/emulator/journeys.

## Global Constraints

- **NEVER commit or push** (CLAUDE.md — overrides the commit steps this skill normally requires). Each task ends with a verification step instead; the user reviews and commits.
- Execute in an isolated worktree (`superpowers:using-git-worktrees`) created at execution start.
- **Version pins are fixed by this plan** (researched 2026-07-15). Do not "helpfully" bump any of them. Kotlin stays 2.3.x: KSP has no Kotlin-2.4-compatible release (google/ksp#2965 open; latest KSP release is 2.3.10).
- `applicationId` / base package: `com.shayanaryan.chatbot`. Launcher label: **Bro**.
- compileSdk 37, targetSdk 37, minSdk 31 — in **every** module. 37 is not optional: `androidx.core:core-ktx` 1.19.0 and the `androidx.lifecycle` 2.11.0 artifacts hard-require compileSdk ≥ 37 via AGP's AAR-metadata check, which fails the build and cannot be suppressed. Do not "correct" these back to 36. JDK 17 toolchain; JDK 21 only as the test-JVM launcher for Robolectric (SDK 36 requirement — Robolectric's emulated SDK stays 36 and is independent of compileSdk; 4.16.1 does not support 37).
- AGP 9 built-in Kotlin: never apply `org.jetbrains.kotlin.android`; set `compileOptions.targetCompatibility` once per module — `jvmTarget` inherits it. Never add a `composeCompiler {}` block.
- Configuration cache ON; Isolated Projects OFF.
- Module dependency rules (architecture skill, hard): `:feature:*` → `:shared` + `:core:ui` only; `:core:ui` → nothing in project; `:shared` → nothing in project; `:app` → everything. Hilt never inside `:shared`. `commonMain` never imports `android.*`.
- No mocking libraries. Navigation 2 (`navigation-compose`) prohibited (don't add any nav dependency in M0 — Nav 3 is pinned in the catalog for M1, unused).
- **Template output is ground truth for AGP 9 DSL.** The build-file code blocks below are modeled on the `android create` template; where the generated template's Kotlin/JVM/Compose wiring differs syntactically, keep the template's form and only change values (versions, namespace, SDK levels). When unsure about an API, use `android docs search "<topic>"` — never guess from memory.
- The `android` CLI is installed (`~/.local/bin/android`). If any subcommand's flags are unclear, run `android <cmd> --help` first.
- **Formatting is Spotless + ktlint 1.8.0, `ktlint_official`, configured in root `.editorconfig`** (4-space indent, `max_line_length = 100`). Code blocks in this plan are indicative, not byte-exact: write the code, then run `./gradlew spotlessApply` and let it settle the layout. `assembleDebug` does not run Spotless; root `check` does, so a module that only ran `assembleDebug` can still fail the gate. ktlint also lints — it rejects code the plan's own blocks contain (long lines, misnamed files), and those are real findings, not noise.

`Theme.kt` — static palette by default, dynamic color opt-in:

```kotlin
package com.shayanaryan.chatbot.core.ui.theme

import androidx.compose.foundation.isSystemInDarkTheme
import androidx.compose.material3.MaterialTheme
import androidx.compose.material3.darkColorScheme
import androidx.compose.material3.dynamicDarkColorScheme
import androidx.compose.material3.dynamicLightColorScheme
import androidx.compose.material3.lightColorScheme
import androidx.compose.runtime.Composable
import androidx.compose.ui.platform.LocalContext

private val DarkColorScheme =
  darkColorScheme(primary = Purple80, secondary = PurpleGrey80, tertiary = Pink80)

private val LightColorScheme =
  lightColorScheme(primary = Purple40, secondary = PurpleGrey40, tertiary = Pink40)

@Composable
fun ChatbotTheme(
  darkTheme: Boolean = isSystemInDarkTheme(),
  dynamicColor: Boolean = false,
  content: @Composable () -> Unit,
) {
  val colorScheme =
    when {
      dynamicColor -> {
        val context = LocalContext.current
        if (darkTheme) dynamicDarkColorScheme(context) else dynamicLightColorScheme(context)
      }
      darkTheme -> DarkColorScheme
      else -> LightColorScheme
    }

  MaterialTheme(colorScheme = colorScheme, typography = Typography, content = content)
}
```

Both color paths exist from M0 so spec 002 changes palette and type *values*, not theme structure. `dynamicColor` needs no SDK guard: minSdk 31 is the API level dynamic color arrived in. `Typography`'s `bodyLarge` currently restates the M3 default, so it is behaviourally a no-op — it exists as the anchor point 002 edits.
